Domain Overview

Literature survey, research gap, problem statement, objectives, methodology and technologies.

Literature Survey

The rapid expansion of IoT has transformed sectors from manufacturing and healthcare to energy and transportation. These devices require frequent firmware updates, making secure Over-the-Air (OTA) delivery a critical research area. The following key works inform this project.

SecFOTA - Saad El Jaouhari (2022)

Targets insecure OTA updates on highly constrained IoT devices. Defines roles (Manufacturer, Supplier, Gateway, IoT Device) and a trust chain using public-key signatures and secure channels. Includes mutual authentication and an attestation step where the IoT device sends proof of a successful update back to the gateway. Limitation: does not implement dynamic access policies or comprehensive audit logging, leaving it unable to respond to runtime changes in device trust status.

ASSURED - N. Asokan et al. (2018)

Presents a secure firmware update framework for large-scale IoT deployments. Introduces an authorization token bound to each firmware package, where the manufacturer signs an update and includes a token encoding device-specific constraints, effectively addressing end-to-end authenticity. Limitation: its static token model lacks per-request policy flexibility and provides no mechanism for detecting changes in device hardware state between update cycles. No audit logs or continuous attestation during operation are provided

Zero Trust MQTT IoT - Meha James et al. (2025)

Proposes a Zero Trust architecture for general IoT networking, specifically MQTT messaging. Embeds an ABAC engine inside the MQTT broker as a Policy Enforcement Point, adding only approximately 20–45ms of latency versus over 100ms for traditional setups. Limitation: targets the communication layer rather than firmware updates, so it does not address firmware authenticity or device attestation at the OTA delivery level.

DICE-Based Runtime Attestation- Stefan Hristozov et al. (2018)

Presents an on-device runtime attestation scheme for tiny IoT microcontrollers, isolating an attestation routine in privileged mode using a standard MPU and lockable ROM secret. The MCU executes isolated code to measure its firmware hash and returns a signed report, making it feasible on low-cost devices without TPM or TrustZone. Limitation: only generates attestations without addressing how updates are fetched, authorized, or logged.

Research Gap

Current Over-the-Air (OTA) firmware update frameworks for IoT devices, including widely used platforms like Mender, Eclipse Hawkbit, Uptane, and TUF focus almost entirely on securing the delivery of firmware through encryption and integrity verification. However, they share several critical blind spots:

  • No device state validation before updates are applied - compromised or unstable devices can still receive firmware
  • Static trust models - once a device is enrolled, it is trusted indefinitely, even if it becomes physically tampered with or behaviorally compromised.
  • Passive recovery only - standard hardware watchdog timers only react after a complete system crash, missing subtle threats like memory leaks or thread starvation
  • No integration between anomaly detection and update decisions — monitoring systems exist separately and don't influence whether an update proceeds.
  • Heavy ML-based detection - existing anomaly detection approaches are computationally too expensive for resource-constrained IoT devices.
  • Limited audit trails - most frameworks provide minimal logging, reducing forensic and compliance capability

What This Work Addresses

  • Layer 1 : Zero Trust Authorization introduces a five-phase adaptive authentication pipeline that continuously re-verifies every device before each update transaction. Unlike static enrollment models, this layer uses hardware-bound cryptographic identity, adaptive trust scoring, and dynamic role and policy-based access control to ensure only genuinely trusted devices receive firmware.
  • Layer 2 : Lightweight Secure OTA Core addresses the inefficiency of full firmware transfers by introducing an instruction-aware delta patching system. Rather than blindly applying delta patches, a decision layer evaluates four factors - Structural Change Ratio, Fragmentation Score, Reconstruction Complexity, and Instability Risk to determine whether a delta or full update is more appropriate. An optional peer-assisted mesh distribution model further reduces server load.
  • Layer 3 : Threshold-Based Anomaly Detection tackles the lack of pre-update device health validation by continuously monitoring telemetry parameters such as CPU usage, memory consumption, battery level, and network performance. A lightweight rule-based detection engine gates firmware updates, blocking them when a device exhibits anomalous behavior — without the overhead of machine learning models.
  • Layer 4 : Resilient Fail-Safe Update Engine addresses the gap in post-update behavioral verification. Rather than relying on passive watchdog timers, a custom Finite State Machine (FSM) actively monitors stack memory, heap depletion, and peripheral stability during a probation period after each update, triggering an autonomous rollback in under 45ms if faults are detected.

Research Problem

IoT OTA update systems are fundamentally reactive and trust-blind. They secure the firmware package in transit but ignore whether the receiving device is trustworthy, stable, or behaving normally before, during, or after the update. This creates a chain of vulnerabilities: unauthorized devices can receive updates, anomalous devices can have updates forced upon them, corrupted payloads can survive cryptographic checks, and faulty firmware can persist undetected after installation.

Objectives

Main Objective

To design and implement a modular, secure, and resilient OTA firmware update framework for IoT environments that enforces continuous trust verification, lightweight anomaly detection, efficient firmware delivery, and autonomous post-update recovery.

Specific Objectives

  • Design a five-phase Zero Trust authentication pipeline that binds device identity to physical hardware characteristics and produces a quantitative trust score per update transaction.
  • Implement dynamic RBAC/PBAC policy enforcement and a nine-category tamper-evident audit logging system for full forensic traceability.
  • Develop a decision-driven delta patching engine that evaluates firmware change characteristics before choosing between delta and full updates, reducing bandwidth and transmission risk.
  • Build a lightweight threshold-based anomaly detection engine that monitors real-time device telemetry and gates OTA updates based on device health status
  • Engineer an active hardware attestation system using a FreeRTOS-based FSM on ESP32 that detects execution faults during a post-update probation window and triggers autonomous rollback before system failure.

Methodology

The SecureTrustOTA framework follows a modular, layered system engineering approach, where each layer is independently designed and implemented, then integrated into a unified pipeline. The overall flow moves from authorization → optimized delivery → pre-update health check → post-update verification.

Layer 1 : Zero-Trust Authorization Layer

Layer 1 operates as a Flask-based server with a SQLite backend. Devices progress through five authentication phases - cryptographic identity, hardware binding, challenge-response, context-bound proof, and adaptive trust scoring before any update is authorized. The trust score is computed entirely server-side, preventing compromised devices from self-reporting fraudulent scores.

Flow: Authenticate → Authorize → Verify → Log & Audit

Layer 2 : Lightweight Secure OTA Core

Layer 2 introduces a pre-delivery analysis stage that computes a Decision Score from four firmware change metrics. If the score is low, an instruction-aware delta patch is generated by parsing the firmware's ELF structure and normalizing address references before diffing. High scores trigger a full firmware transfer instead. An optional mesh distribution model allows peer devices to relay updates, reducing central server load.

Flow: Collect → Generate & Encrypt → Verify & Apply → Validate / Rollback → Alert & Monitor

Layer 3 : Threshold-Based Anomaly Detection Engine

Layer 3 runs a continuous telemetry monitoring loop on the OTA server. Device metrics are compared against predefined and adaptive threshold values using rolling window analysis. A debounce mechanism prevents transient spikes from triggering false positives. If a device is flagged as anomalous, the OTA gating mechanism blocks the update and generates an alert on the monitoring dashboard.

Flow: Telemetry → Compare Thresholds → Update (if normal) → Alert & Monitor (if abnormal)

Layer 4 : Resilient Fail-Safe Update Engine

Layer 4 is implemented directly on the ESP32 hardware using the ESP-IDF framework. Incoming firmware is first written to an SD card buffer (air-gapped staging) rather than directly to flash, preventing corruption during download. After SHA-256 streaming verification, the firmware is written to the inactive partition. Upon reboot, a high-priority FreeRTOS attestation task monitors heap memory, stack usage, and peripheral connectivity for 30 seconds. If abnormalities are detected, the FSM marks the partition invalid and forces a rollback to the stable partition.

Flow: Update → Verify → Test → Rollback if fail → Keep running if pass
